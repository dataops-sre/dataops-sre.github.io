# Repository Writing Taxonomy Guide

This file defines the recommended **category** and **tag** taxonomy for future posts in this repository.

The goal is to keep front matter:
- consistent
- searchable
- easy to reuse
- easy to browse in Jekyll archives and topic pages

---

## Core rules

### Categories
Categories should be treated as **single units**.

- Use **single-word categories** when possible
- If a category concept contains multiple words, join them with `-`
- Do **not** use spaces inside a category name

### Good category examples
- `SRE`
- `DevOps`
- `Kubernetes`
- `Data-Engineering`
- `Cloud-Infrastructure`
- `Developer-Experience`
- `GitHub-Pages`

### Bad category examples
- `Data Engineering`
- `Cloud Infrastructure`
- `Developer Experience`

---

### Tags
Tags can be more human-readable.

- Spaces are allowed in tags
- Prefer natural wording
- Reuse existing tags whenever possible
- Avoid overly vague tags like `General`

### Good tag examples
- `Data Engineering`
- `Data Processing`
- `Cloud Infrastructure`
- `Developer Experience`

## Tagging philosophy

### Categories are broad
Categories should represent the main area a post belongs to.

Examples:
- a Spark on Kubernetes article belongs to `Data-Engineering` and `Kubernetes`
- a Docker runtime debugging article belongs to `SRE`, `Docker`, and `Linux`
- a Jekyll publishing article belongs to `Blogging` and `GitHub-Pages`

### Tags are specific
Tags should capture:
- technologies
- problem domains
- techniques
- recurring themes

Examples:
- `Apache Spark`
- `Data Processing`
- `Monitoring`
- `Troubleshooting`
- `Cloud Infrastructure`

---

## Recommended limits per post

To keep front matter clean:

- use **2 to 4 categories** per post
- use **4 to 8 tags** per post

If a post needs more than that, the taxonomy is probably too broad or too noisy.

---

## Preferred distinctions

### `SRE` vs `DevOps`
Use:
- `SRE` for reliability, monitoring, runtime debugging, operational issues, production behavior
- `DevOps` for tooling, platform setup, infrastructure, deployment workflows

A post can have both, but not every operational post needs both.

---

### `Data Engineering` vs `Data Processing`
Use:
- `Data Engineering` for the broad domain
- `Data Processing` for actual processing workloads, pipelines, ETL, streaming, batch jobs

Spark and Airflow posts often benefit from both.

---

### `Observability` vs `Monitoring`
Use:
- `Observability` for tracing, metrics, logs, telemetry, broad system visibility
- `Monitoring` for concrete alerting/metric collection/operational monitoring topics

A Datadog post may use both.

---

## Naming conventions

### Capitalization
Keep capitalization consistent with common industry usage.

Preferred forms:
- `DevOps`
- `SRE`
- `Apache Spark`
- `Airflow`
- `Kubernetes`
- `Datadog`
- `GitHub`
- `GitHub Pages`
- `macOS`

Avoid inconsistent forms like:
- `Devops`
- `Spark` when `Apache Spark` is more precise
- `Github`
- `MacOS`

## Practical rule for future posts

Before inventing a new category or tag, ask:

1. Does an existing category already cover this?
2. Can this be a tag instead of a category?
3. Is this term broad enough to be reused later?
4. Is the wording consistent with the rest of the repository?

If not, reuse the closest existing term.

---

## Default recommendation for new posts

When writing a new post:

- choose the **main domain** as categories
- use tags for the **specific technologies and subtopics**
- avoid one-off labels unless the topic is genuinely new

This keeps the repository coherent over time.
